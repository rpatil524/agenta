Agenta's observability layer sits on top of OpenTelemetry. A **span** is a unit of work; a **trace** is a set of spans grouped by the same `trace_id`. Spans carry standard OTel fields plus an `ag.*` attribute subtree that Agenta owns. The `ag.*` subtree is what links traces back to applications, evaluators, testsets, and the other Agenta entities.

This page covers the span and trace model, the two trace types, how references work, the four ingestion paths, the async write contract, the `ag.*` namespace, and querying.

## Spans and traces

A span records one unit of work, such as a request, an LLM call, or a tool execution. Every span carries:

| Field | Description |
|-------|-------------|
| `trace_id` | 32-char hex that groups all spans of one operation. |
| `span_id` | 16-char hex identifying this span. |
| `parent_id` | The parent's `span_id`, or null for a root span. |
| `start_time`, `end_time` | ISO-8601 timestamps. |
| `span_kind` | OTel kind (`SPAN_KIND_SERVER`, `SPAN_KIND_INTERNAL`, and so on). |
| `status_code` | `STATUS_CODE_UNSET`, `STATUS_CODE_OK`, or `STATUS_CODE_ERROR`. |
| `attributes` | Free-form dict. Agenta's data lives under the `ag.*` subtree. |

A trace is the set of spans sharing one `trace_id`. On endpoints that return a single trace, the payload is the `trace_id` plus a nested `SpansTree`: spans keyed by `span_name` under each parent's `spans` field (a list when siblings share a name). This is the shape the UI renders as a waterfall.

## Trace types

Every trace declares a type through `ag.type.trace`:

| Value | Meaning |
|-------|---------|
| `invocation` | The trace records a product event: an app run, an evaluator execution, an LLM call. Most traces produced by instrumented code are invocations. |
| `annotation` | The trace records a human-authored or evaluator-produced judgment on another trace. Annotations carry a `link` back to the invocation they describe. |
| `unknown` | The type was not declared or could not be inferred. |

The distinction matters at query time. Annotation UIs filter for `ag.type.trace = annotation`; evaluation views walk invocations.

Each span also declares its role through `ag.type.span`:

| Value | Role |
|-------|------|
| `workflow` | The root of a product run (an application or evaluator invocation). |
| `agent` | An agent loop driving tool calls or sub-tasks. |
| `chain` | A composed sequence of calls treated as one unit. |
| `task` | A generic unit of work. Default when no other type fits. |
| `tool` | A tool call from an agent. |
| `llm` | A raw LLM request. |
| `chat`, `completion` | Chat-completion or completion calls. |
| `embedding`, `query`, `rerank` | Index and retrieval calls. |

## References

A reference links a trace (or a span) back to the Agenta entity that produced or triggered it. References live under `ag.references.<key>` and each value names an entity revision:

```json
{"id": "019d952f-0000-0000-0000-000000000000", "slug": "support-bot", "version": "3"}
```

Any of the three fields is enough. `id` always resolves. `slug` resolves within the project. `version` applies to revisions and is a string.

The supported keys:

| Key | Points at |
|-----|-----------|
| `application`, `application_variant`, `application_revision` | The application artifact, variant, or revision. |
| `workflow`, `workflow_variant`, `workflow_revision` | The workflow artifact, variant, or revision. |
| `evaluator`, `evaluator_variant`, `evaluator_revision` | The evaluator artifact, variant, or revision. |
| `testset`, `testset_variant`, `testset_revision` | The testset artifact, variant, or revision. |
| `query`, `query_variant`, `query_revision` | A saved query, variant, or revision. |
| `environment`, `environment_variant`, `environment_revision` | A deployment target. |
| `snippet`, `snippet_variant`, `snippet_revision` | A reusable snippet. |
| `testcase` | A specific testcase. |

## Links vs references

The two terms look similar but do different things.

|  | Links | References |
|---|-------|------------|
| Spec | OTel `SpanLink` | Agenta extension |
| Points at | Another span in another trace (`trace_id` + `span_id`) | An Agenta entity (`id` / `slug` / `version`) |
| Used for | "This span continues from span Y in that other trace." | "This trace was produced by application revision X." |

An annotation trace typically carries both: a `link` to the invocation span being annotated, and a `reference` to the evaluator that produced the annotation.

## How evaluations weave traces together

Evaluations tie traces and the rest of the API together:

1. Each evaluation run invokes its subject (an application revision) over a testset.
2. Every invocation produces an `invocation` trace.
3. Every evaluator execution produces another `invocation` trace.
4. Evaluation **scenarios** reference those trace IDs via their links.

So a single evaluation run leaves behind one invocation trace per (testcase × subject) plus one invocation trace per (testcase × evaluator). See [Evaluations](/reference/api-guide/evaluations) for how scenarios join them back together.

## Ingestion

Tracing has four write endpoints. Pick the one that matches your payload shape.

| Endpoint | When to use |
|----------|-------------|
| `POST /otlp/v1/traces` | Raw OTLP protobuf from an OTel SDK or collector. Use this when your existing OTel instrumentation can just point its exporter at Agenta. |
| `POST /tracing/spans/ingest` | Agenta-native structured ingest. Accepts either a flat `spans` list (with `parent_id` linking) or a nested `traces` tree. Use this when you're writing spans yourself with full control over IDs and hierarchy. |
| `POST /simple/traces/` | Single-trace helper for feedback or annotations. The endpoint generates the IDs and wraps the payload. Use this for one standalone trace. |
| `POST /traces/ingest`, `POST /traces/`, `PUT /traces/{trace_id}` | List-shaped trace CRUD. Accepts the canonical `Trace` and `Traces` shape. Use this when operating on whole traces as records. |

All four paths converge on the same ingest pipeline.

### Async write contract

Every ingest endpoint returns `202 Accepted`, never `200 OK`. The contract:

- The router parses the payload, emits stream messages, and returns. Persistence and full validation run asynchronously in a worker.
- OTel semantics require the ingest buffer to return fast so instrumented apps do not block on telemetry.
- The `count` on the response is the number of spans **parsed into the stream**, not the number the worker persisted.
- If you submit `N` spans and see `count < N`, some spans failed at parse time (malformed JSON, invalid IDs). Worker failures are not visible on the response.

An earlier edge case where validation failures surfaced as `count: 0` was tracked under AGE-3735 and resolved. The ingest path stays stream-accept by design.

## The `ag.*` attribute namespace

Spans carry an OTel `attributes` dict. Agenta owns the `ag` subtree.

| Namespace | Purpose |
|-----------|---------|
| `ag.type.trace`, `ag.type.span` | Trace and span type enums. |
| `ag.data.inputs`, `ag.data.outputs`, `ag.data.parameters`, `ag.data.internals` | What the span saw and produced. |
| `ag.metrics.duration`, `ag.metrics.errors`, `ag.metrics.tokens`, `ag.metrics.costs` | Server-computed metrics. Each carries a `cumulative` and `incremental` view. |
| `ag.references` | Entity references (see above). |
| `ag.flags`, `ag.tags`, `ag.meta` | Filtering and labelling dicts. |
| `ag.exception` | Structured exception payload. |
| `ag.session.id`, `ag.user.id` | Optional session and user identifiers. |

Two wire formats are accepted on ingest. Query responses always return the nested form.

**Flat (dot-notation):**

```json
{
  "attributes": {
    "ag.type.trace": "invocation",
    "ag.type.span": "workflow",
    "ag.data.inputs.country": "France",
    "ag.data.outputs": "Paris"
  }
}
```

**Nested:**

```json
{
  "attributes": {
    "ag": {
      "type": {"trace": "invocation", "span": "workflow"},
      "data": {"inputs": {"country": "France"}, "outputs": "Paris"}
    }
  }
}
```

### `ag.unsupported` fallback

Agenta does not reject unknown or malformed `ag.*` fields. The ingest pipeline:

1. Moves unknown keys, wrong-typed values, and unparseable JSON strings under `ag.data.{inputs, parameters, internals}` into `ag.unsupported.*`.
2. On a Pydantic validation error, relocates the offending top-level key into `ag.unsupported` and retries.
3. On a second failure, falls back to a minimal valid attribute object with everything in `unsupported`.

This is a product commitment: keep customer telemetry, even when fields don't validate. If you see fields under `ag.unsupported.*` after ingest, the worker accepted the trace and parked those fields for you to inspect.

## Querying

`POST /tracing/spans/query` takes filtering, windowing, and a `focus`:

- `focus = "trace"` (default). Response is a nested `traces` tree keyed by `trace_id` → span name. The same shape that `spans/ingest` accepts under `traces`.
- `focus = "span"`. Response is a flat `spans` list. Paginates without rebuilding hierarchies.

Exactly one of `traces` or `spans` is populated. `POST /spans/query` and `POST /traces/query` wrap the same backend, forcing `focus=SPAN` and `focus=TRACE` respectively. Use [Windowing](/reference/api-guide/query-pattern#windowing) for cursor pagination.

## Examples

### Ingest a nested trace

```bash
curl -X POST "$AGENTA_HOST/api/tracing/spans/ingest" \
  -H "Content-Type: application/json" \
  -H "Authorization: ApiKey $AGENTA_API_KEY" \
  -d '{
    "traces": {
      "f5a2efb40895881e938e2ebc070beca8": {
        "spans": {
          "workflow_v0": {
            "trace_id": "f5a2efb40895881e938e2ebc070beca8",
            "span_id": "15f3df0731995245",
            "span_kind": "SPAN_KIND_SERVER",
            "start_time": "2026-04-16T18:18:18.491929Z",
            "end_time": "2026-04-16T18:18:20.415372Z",
            "attributes": {
              "ag": {
                "type": {"trace": "invocation", "span": "workflow"},
                "data": {"inputs": {"country": "France"}, "outputs": "Paris"}
              }
            },
            "spans": {
              "llm_call": {
                "trace_id": "f5a2efb40895881e938e2ebc070beca8",
                "span_id": "c8ae7f12d1b3e9a4",
                "parent_id": "15f3df0731995245",
                "start_time": "2026-04-16T18:18:18.800000Z",
                "end_time": "2026-04-16T18:18:20.400000Z",
                "attributes": {"ag.type.span": "llm"}
              }
            }
          }
        }
      }
    }
  }'
```

Response (`202 Accepted`):

```json
{
  "count": 2,
  "links": [
    {"trace_id": "f5a2efb40895881e938e2ebc070beca8", "span_id": "15f3df0731995245"},
    {"trace_id": "f5a2efb40895881e938e2ebc070beca8", "span_id": "c8ae7f12d1b3e9a4"}
  ]
}
```

### Record a single annotation

```bash
curl -X POST "$AGENTA_HOST/api/simple/traces/" \
  -H "Content-Type: application/json" \
  -H "Authorization: ApiKey $AGENTA_API_KEY" \
  -d '{
    "trace": {
      "origin": "human",
      "kind": "adhoc",
      "channel": "api",
      "data": {"outputs": {"score": 5, "comment": "Good response"}},
      "references": {"evaluator": {"slug": "user-feedback"}},
      "links": {
        "invocation": {
          "trace_id": "0af7651916cd43dd8448eb211c80319c",
          "span_id": "b7ad6b7169203331"
        }
      }
    }
  }'
```

The response returns the created trace with a server-generated `trace_id` and `span_id`. Use those IDs with `GET`, `PATCH`, or `DELETE` on `/simple/traces/{trace_id}`.
